### 暗黙の前提・仮説
- クラウドAI（ChatGPT等）では制限されるユースケースを想定している可能性がある
- 「2B」という小さいモデルサイズを意識して選択している（軽量・高速優先か）

## 4. リサーチ結果

### 論点1: `jaahas/qwen3.5-uncensored`はどのような技術的手法で無検閲化されているか？その品質劣化とリスクは？

#### ファクト
- `jaahas/qwen3.5-uncensored`はOllamaレジストリで配布されているコミュニティ作成モデル。2b/4b/9b/27bのサイズ展開があり、いずれも256Kトークンのコンテキストウィンドウを持つ [1]
- 「無検閲」の実装手法は主に**Abliteration（アブリタレーション）**と呼ばれる技術による。有害プロンプトと無害プロンプトへの活性化の差分からモデルの「拒否方向（refusal direction）」を特定し、残差ストリームから外科的に除去する手法 [2][3]
- abliterationのベンチマーク比較研究（2025）では、最良の手法（ErisForge）でGSM8K数学推論スコアが平均-0.28ポイントの劣化にとどまる。ただし数学的推論は最も影響を受けやすいタスク [4]
- 安全フィルタ除去後の有害リクエスト応答率：通常モデル18.8% → 無検閲版74.1% [5]
- 同種の競合モデルとして `huihui_ai/qwen3.5-abliterated`（OllamaとHuggingFaceで配布）、`HauhauCS/Qwen3.5-2B-Uncensored`などが存在する [6]

#### 分析・考察
Abliterationは「モデルの安全フィルタは特定の低次元部分空間で実装されている」という発見に基づく手術的介入であり、通常使用での言語能力はほぼ維持される。ただし、能力劣化が「最小」であるという研究結果は、ベストな実装ツールを使った場合の話であり、jaahasが具体的にどのツール・パラメータを使用したかは不明。HuggingFaceにソース情報があるとされているが、詳細なメタデータは確認できなかった。

複数のコミュニティ作成者が同じベースモデルを無検閲化している競争状況は、技術の習得コストが低下していることを示す。一方で、コミュニティ版はAlibaba公式の品質保証外であるため、本番用途での信頼性評価は自分で行う必要がある。

#### 小括
`jaahas/qwen3.5-uncensored`はAbliteration手法でQwen3.5の安全フィルタを除去したコミュニティモデル。通常タスクでの能力劣化は軽微だが、有害応答率は大幅に上昇する。技術的に安定した手法だが、作成者の実装詳細は不透明。

### 論点2: Qwen3.5-2Bの実力はどの水準か？ローカル実行で体験品質を左右するボトルネックは何か？

#### ファクト
- Qwen3.5は2026年3月2日にAlibabaのQwenチームがリリース。スモールシリーズは0.8B/2B/4B/9Bで構成 [7]
- Qwen3.5-2Bはネイティブマルチモーダル（テキスト・画像・動画）、248K語彙、201言語対応、256Kトークンコンテキスト対応 [7][8]
- モデルの設計思想は「モバイル・エッジデバイス向け」。電池寿命を重視しオフライン動作を想定 [8]
- ハードウェア要件: 2Bモデルは2～4GB VRAMのGPUで動作可能。CPUのみの場合は7～12 tokens/秒程度（現代的CPUで）、GPU（RTX 4060/8GB VRAM）なら快適なインタラクティブ速度 [9]
- Qwen3.5上位モデル（27B）のベンチマーク：MMLU-Pro 86.1、GPQA Diamond 85.5、SWE-bench Verified 72.4。2Bは公開ベンチマークで27Bの約60〜70%水準と推定される [10]
- 2Bモデルのベンチマーク評価時の出力トークン量は約3.9億と、大型モデルより多く、内部推論コストの高さを示す [8]

#### 分析・考察
2Bというパラメータ数は、GPT-4oやClaude Sonnet相当を期待するには力不足だが、「軽量・高速」の文脈では非常に有能。256Kコンテキストは2Bとしては破格であり、長文処理が必要な作業（書類要約、コード解析）で相対的強みを発揮する。

ボトルネックはGPU有無で決定的に分かれる。CPUのみで7～12 tokens/秒はインタラクティブチャットとしてギリギリ許容範囲。GPU搭載Macなら十分な快適性。「ながら作業」ではなくリアルタイム対話目的なら、GPU（Apple M系チップのメモリ統合含む）の存在が体験品質の分水嶺。

#### 小括
Qwen3.5-2BはCPUのみでも動くが体験はGPU有無で大幅に差が出る。長文コンテキストや多言語対応は2Bとしては優秀で、ローカル実行の小型モデルとして実用ラインに届く。

### 論点3: Ollama + Open WebUIスタックはChatGPT代替として実用に堪えるか？その差分と可能性は？

#### ファクト
- Open WebUIは2026年3月時点でGitHubスター90k〜124k以上、ダウンロード2.82億以上。最新バージョンはv0.8.10（2026年3月9日リリース）[11]
- 主要機能：RAG（文書取込みチャット）、音声通話、ビデオコール、Pythonネイティブ関数呼び出し、カスタムエージェントビルダー、成果物の永続ストレージ、SSO・RBACによるエンタープライズ管理 [11]
- Ollamaはローカルのモデルダウンロード・管理・推論を担い、Open WebUIがGUIフロントエンドとして乗る構成。DockerまたはネイティブインストールでOllamaとOpen WebUIを連携させる [12]
- セットアップは`ollama pull jaahas/qwen3.5-uncensored:2b`と`docker run ...open-webui`の2コマンドが基本 [12][13]
- Ollamaモデルレジストリにはjaahas版を含む複数の無検閲Qwen3.5バリアントが存在し、UIからモデル切替が可能

#### 分析・考察
ChatGPTとの最大の差分は「オンライン機能の不在」（ウェブ検索、プラグイン等はOpen WebUI側のアドオン設定が必要）と「モデル性能上限」（2Bはあくまで軽量モデル）。しかし、RAG、音声対話、関数呼び出し、エージェント構成が揃っており、機能面での差は2025〜2026年で急速に縮まっている。

Open WebUIの進化は際立っており、90k→124k+スターという急増と282M+ダウンロードは、ローカルAIのUI層として業界デファクトに近い地位を確立していることを示す。Ollama側もモデル管理の簡便さで圧倒的な支持を得ており、スタックとしての成熟度は高い。

#### 小括
Ollama + Open WebUIは機能面でChatGPTに迫っており、ローカルAIとしての実用性は高い。セットアップの敷居も下がっており、2Bモデルでのチャットボット体験は「軽量版ChatGPT」として充分機能する水準にある。

### 論点4: ローカル無検閲LLMの本質的価値はどこにあるか？どのユースケースで差別化されるか？

#### ファクト
- 日本企業のローカルLLM導入動機：情報セキュリティとデータ主権への意識向上が主因。金融・医療・建設など機密情報を扱う業種での導入事例が増加 [14]
- EUデータ法（2025年9月施行）・EU AI法（2026年8月全面適用）など、データ越境移転の規制が強化される世界的潮流 [15]
- ローカルLLMの利点：入力データと出力テキストが社内/PC外に出ない、オフライン動作、長期的なコスト削減 [14]
- 無検閲モデルのユースケース（正当）：成人向けコンテンツ創作、セキュリティリサーチ（脆弱性の解析・ペンテスト支援）、法律・医療分野でのデリケートなシナリオ検討、多言語翻訳の文化的ニュアンス向上 [16]
- 「会話が外部送信されない」という安心感の下で、クラウドには入力しにくい機密文書の処理が可能になる [14]

#### 分析・考察
無検閲ローカルLLMの価値は「検閲がない」という単一の軸ではなく、「プライバシー × 検閲なし × コストゼロ（API課金なし）」の三要素の組み合わせに本質がある。クラウドモデルは性能で勝るが、①社外秘データ、②倫理的にグレーな調査、③API費用を気にせず大量に叩きたいユースケースでローカルが選択肢になる。

2B程度の性能で事足りるユースケース（テキスト校正、定型文生成、QA、コード補完）なら、無検閲ローカルモデルは「費用ゼロ・データ漏洩リスクゼロ」という強みが際立つ。

#### 小括
ローカル無検閲LLMの本質的価値は「プライバシー保護」と「制約なしの応答」の組み合わせにある。クラウドモデルとは補完関係であり、機密データ処理・検閲に阻まれる正当な調査・コスト削減の3軸で差別化される。

### 論点5: 無検閲ローカルLLMの運用リスクはどこにあるか？法的・倫理的グレーゾーンの実態は？

#### ファクト
- 無検閲LLMの有害リクエスト応答率は、通常モデルの18.8%から74.1%に上昇。詐欺、心理操作、物理的危害、社会的混乱を引き起こし得る応答を含む [5]
- Cisco Talosの報告（2025）：サイバー犯罪者が無検閲LLMを犯罪活動の補助に積極利用しており、カスタム構築の犯罪者向けLLMも出現 [17]
- MDPI掲載の研究（2025）：一般公開・ローカル配備可能な無検閲LLMの追跡調査。技術的アクセスの容易化が進む一方、安全ガバナンスの整備が追いついていない [5]
- 日本のAI安全ガバナンス：経産省がAI安全研究所（J-AISI）を2024年2月に設置。ただし無検閲LLMのローカル実行を直接規制する法律は現時点で日本では未整備 [18]
- 一般的な法的リスク：生成されたコンテンツの内容によって、刑法・著作権法・個人情報保護法・不正アクセス禁止法などが適用される可能性（ツール自体ではなく、生成コンテンツの使用が問われる）

#### 分析・考察
「ローカルで動かす」という行為自体は現状日本では合法。問題はモデルが生成したコンテンツの使用にある。違法なコンテンツを生成させて使用・公開すれば、モデルではなくユーザーが法的責任を負う構造。無検閲モデルはその「ガードレールなし」という特性上、ユーザーの倫理意識と使用目的が従来以上に重要になる。

研究者・セキュリティ専門家・創作者など正当なユースケースは存在するが、「無検閲だから何でもできる」という誤解は危険。悪用リスクの現実を踏まえた上で、明確な使用目的を持って運用することが前提条件。

#### 小括
ローカル実行自体は現状日本で違法ではないが、生成コンテンツの性質と用途が法的・倫理的責任を決める。無検閲モデルはリスクが実質的に使用者側に転嫁されるモデルであり、使用目的の明確化が不可欠。

## 5. まとめ

### 総合所見
`jaahas/qwen3.5-uncensored:2b` × Ollama + Open WebUIは、「軽量・プライベート・制約なし」という三軸を同時に実現するローカルAI環境であり、技術的成熟度・セットアップの容易さ共に2026年時点で実用水準に達している。2Bモデルの性能的限界を理解した上で適切なユースケースに絞れば、クラウドモデルに依存しない有効な代替になりうる。

### 確度の高い結論
- Ollama + Open WebUIスタックはセットアップが容易で、機能的に成熟しており、ローカルAIのUI基盤として現時点のデファクト（90k〜124k GitHub Stars）
- Abliteration手法による無検閲化の能力劣化は軽微（ベスト実装で-0.28 pt）だが、有害応答率は18.8%→74.1%と大幅上昇する
- ローカル処理であれば機密データを外部に送出しないため、データ主権の観点での合理的選択肢になる
- ローカル実行自体の違法性は現状日本では認められていないが、生成コンテンツの内容・用途によって法的責任がユーザーに帰属する

### まだ不確実な点
- `jaahas`版が具体的にどのabliterationツール・パラメータで作成されたかの詳細（公式ドキュメント不在）
- Qwen3.5-2B単体の公式ベンチマーク数値（スモールシリーズの発表が直近のため詳細データ未公開）
- 日本でのローカル無検閲LLM利用に関する司法・行政解釈（前例なし）
- 長期運用時のモデル更新頻度とコミュニティの継続性

### ネクストアクション
- ハードウェアを確認し、GPU（Apple Siliconの統合メモリ含む）がある場合は2B→4Bへのアップグレードを検討（体験品質が大幅向上する可能性）
- 実際に `ollama run jaahas/qwen3.5-uncensored:2b` を実行し、対象ユースケースでの応答品質を自分でベンチマークする
- 使用目的を文書化する（正当なユースケースの明確化は法的・倫理的観点で重要）
- Open WebUIのRAG機能を活用して、社内ドキュメントとの組み合わせを試す
