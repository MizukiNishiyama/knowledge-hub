# 会議DX ミッション

## テーマ

**会議のDX（デジタルトランスフォーメーション）** — 会議中のリアルタイムAI支援により、意思決定の質を根本から変える

---

## 背景・課題認識

### 会議の本質的価値

会議とは、会社内における**意思決定の場**である。会議の質向上や効率化は、会社の戦略や経営に直接関わる部分の質向上につながり、不可欠な取り組みである。

### 会議の2類型と属人化の問題

会議には大きく分けて以下の二通りがある。

1. **社外の会議（商談・営業等）**
   - ヒアリングや提案など、営業スキルが**属人化**している
   - トップセールスマンと一般のセールスマンで売上の差が**数十倍**開くことも珍しくない

2. **社内の会議**
   - 5人程度のチーム内会議であっても、メンバーの時給が4,000円であれば、1時間の会議だけで**数万円のコスト**がかかる
   - その質が低ければ、直接的なコスト損失となる

### AIエージェント時代における会議のボトルネック

昨今、AIエージェントやAIへの投資が加速している。AIネイティブな世界では、MCP等のプロトコルを通じて各データベースが接続され、最適なデータのやりとりが行える状況が見えてきている。

しかし、そのなかで**会議がボトルネック**になる可能性が高い。

- 意思決定権は当面、人間が持つことが想定される
- 会議で扱われる方向性決定や重要事項の判断をAIに委ねることは、現状では文化的に浸透していない
- つまり、**AIがどれだけ高度に業務を自動化しても、最終的な意思決定の場である会議の質が低ければ、組織全体のアウトプットにキャップがかかる**

### なぜ今か

1. **議事録AIのコモディティ化**: 文字起こし+要約は2025年時点で無料〜低価格で提供されており、もはや差別化にならない。次のフェーズに進むべき時が来ている
2. **ローカルAIの実用化**: Qwen3.5（2026年2月リリース）の9Bモデルは、GPT-OSS-120Bを上回る推論性能をラップトップ上で実現。whisper.cppによるローカル音声認識もリアルタイム処理が実用レベルに達した。**クラウドに依存せず、ローカルマシン上で会議のリアルタイム構造化が技術的に可能になった**
3. **ビッグテックのAI会議機能は「文字起こし+要約」で停滞**: Google Meet (Gemini)、Microsoft Teams (Copilot)、Zoom (AI Companion) いずれも議事録生成が主軸であり、会議中のリアルタイム論点構造化には踏み込んでいない

### 既存ソリューションの限界

- AI議事録ツール（Otter.ai、tl;dv、Rimo Voice等）が登場しているが、これは**ミーティング後のまとめ**にとどまっている
- SuperInternのLive Noteのようにリアルタイム要約を行うツールも出始めているが、「議論の論点を構造化し、未回収の議題を掘り起こす」レベルには達していない
- 会議の質向上や効率化に本質的に活用されているケースは、大企業を含めて**ほとんどない**

## 我々の提案

### コンセプト

**会議中に会議の質を高めるOSSツール** — AIが会議にリアルタイムで伴走し、議論の構造化・論点管理・軌道修正を行う

### 提供する価値

会議のリアルタイム支援として、以下の機能を通じて会議の高品質化を実現する。

| 機能 | 説明 |
|------|------|
| **リアルタイム論点構造化** | 会議の流れをリアルタイムで構造化し、論点のツリーや進捗を可視化する |
| **アジェンダ自動管理** | AIが発言からアジェンダを自動生成・追跡し、事前作成の手間を不要にする |
| **未回収論点の掘り起こし** | 回収されていない議題や質問を検出し、会議中に提示する |
| **議論の軌道修正支援** | 議論の脱線や論理の飛躍を検知し、本筋への回帰を促す |

### なぜOSSか

- **エンタープライズのセキュリティ懸念への回答**: 会議音声データの外部送信に対する抵抗は極めて強い。OSSであればオンプレミス / VPC内での自社運用が可能
- **透明性と信頼**: 会議の内容を扱うツールだからこそ、コードの透明性が導入の信頼につながる
- **コミュニティによる多言語・多文化対応**: 日本語の敬語・曖昧表現への対応など、グローバルコミュニティの知見を活用できる

## 競争優位性

### なぜビッグテックに負けないか — プラットフォーム横断のデータ統合

ビッグテック各社（Google、Microsoft、Zoom）は自社プラットフォーム内でのみAI会議機能を提供している。

しかし、**実際の企業では複数の会議ツールを併用している**のが現実である。

- 社内会議はMicrosoft Teams
- クライアントとの会議はZoom
- カジュアルな打ち合わせはGoogle Meet
- 急なハドルはSlack

このように分散した会議データを**横断的に蓄積・構造化・分析できるのは、プラットフォームに依存しないサードパーティツールだけ**である。

| 観点 | ビッグテック | 本ツール |
|------|------------|---------|
| 対応プラットフォーム | 自社サービスのみ | Zoom / Meet / Teams / Slack 横断 |
| データの統合 | 自社内で閉じる | 全会議を統合データベース化 |
| 会議横断の分析 | 不可能 | 過去の会議文脈を踏まえた論点管理が可能 |
| カスタマイズ性 | 限定的 | OSS + 自社運用で柔軟に対応 |

この**プラットフォーム横断のデータ蓄積**がスイッチングコストとデータモートになり、ビッグテックが個別に機能を追加しても代替できない構造的優位を生む。

### 技術的アプローチ — 完全ローカル処理のデスクトップアーキテクチャ

#### 先行事例: AquaVoice — デスクトップローカル音声処理の成功モデル

[AquaVoice](https://aquavoice.com) は、**Y Combinator W24バッチに採択**されたデスクトップ音声入力ツールである。Product Huntの運営責任者が「今週のコメントはすべてAquaVoiceで書いた」と公言し、TEDに登壇した著名起業家が「自分が見てきた中で最高のデモの一つ」と評するなど、急速にユーザーを拡大している。

AquaVoiceの技術的成功は、「音声処理をローカルで行う」という設計判断にある。

- **独自モデル「Avalon」**: 技術用語（kubectl、PyTorch等）で**97.4%の正答率**を達成。汎用STTモデルのWhisper Large v3（65.1%）やNVIDIA Canary 1B（51.5%）を大幅に上回る
- **450msの低レイテンシ**: ローカル処理により、発言からテキスト表示まで450ms。競合の平均（850ms以上）の約半分
- **完全プライバシー**: 音声データはサーバーに保存されず、ローカルで処理される

**しかし、AquaVoiceが実現したのは「ディクテーション（口述入力）」であり、単一話者の音声をテキストに変換するタスクである。** 本プロダクトが取り組むのは、これを遥かに超える技術的チャレンジ——**複数話者の会議音声をリアルタイムで構造化し、論点を抽出・管理する**というタスクである。この技術的難易度の高さ自体が参入障壁となる。

#### 技術スタック — ローカルLLM + ローカルSTTによる完全オンデバイス処理

本プロダクトは、クラウドAPIに依存しない**完全ローカル処理**を実現する。

**ローカルSTT: whisper.cpp**
- OpenAI Whisperモデルの高性能C++実装（llama.cppと同じ開発者による）
- CUDA / Metal / Vulkan等のGPUアクセラレーション対応
- VAD（音声区間検出）統合によりリアルタイムストリーミング処理が可能
- Whisper Large V3 Turboは元モデルの6倍の推論速度を達成しつつ、精度低下は1-2%以内

**ローカルLLM: Qwen3.5（2026年2月リリース）**
- Alibaba Cloudが開発したオープンウェイトモデル（Apache 2.0ライセンス）
- **Qwen3.5-9B**: わずか9Bパラメータで、120BパラメータのGPT-OSS-120Bを上回る推論性能（GPQA Diamond: 81.7 vs 80.1）
- Gated DeltaNet（線形アテンション）とスパースMoEのハイブリッドアーキテクチャにより、ラップトップ上で実用的な速度で動作
- 4bit量子化で8GB VRAMのGPUで動作可能 — 一般的なビジネスラップトップで実行できる
- ollama / llama.cpp / MLX等のローカル推論フレームワークで即座にデプロイ可能
- ネイティブマルチモーダル対応（テキスト+画像+動画）、201言語対応

```
┌─────────────────────────────────────────────────────┐
│  デスクトップアプリ（完全ローカル処理）                    │
│                                                       │
│  ┌──────────┐    ┌──────────────────────────┐        │
│  │ 音声取得   │ → │ whisper.cpp               │        │
│  │ (会議音声) │    │ ローカルSTT + 話者分離      │        │
│  └──────────┘    └──────────┬───────────────┘        │
│                              │ テキスト（ローカル内）    │
│                              ▼                        │
│  ┌──────────────────────────────────────────┐        │
│  │ Qwen3.5（ローカルLLM）                     │        │
│  │ 論点構造化・アジェンダ管理・軌道修正検知       │        │
│  └──────────────────┬───────────────────────┘        │
│                      ▼                               │
│  ┌──────────────────────────────────────────┐        │
│  │ リアルタイムUI（論点ツリー・進捗・アラート）    │        │
│  └──────────────────────────────────────────┘        │
│                                                       │
│  ※ 音声もテキストも一切外部に送信しない                    │
└─────────────────────────────────────────────────────┘
```

**完全ローカルアーキテクチャの利点:**

- **完全なプライバシー**: 音声データもテキストデータも一切外部に送信されない。エンタープライズのセキュリティ要件を根本的にクリア
- **低レイテンシ**: ネットワーク往復が不要。STT→LLM→UIのパイプラインがすべてローカルで完結
- **プラットフォーム非依存**: デスクトップアプリとしてシステム音声を取得するため、Zoom / Meet / Teams / Slack のいずれでも動作。会議プラットフォームのAPI制約に依存しない
- **ゼロランニングコスト**: クラウドAPI課金が発生しないため、ユーザーの利用量に関わらずインフラコストがゼロ。SaaS事業としての粗利率が極めて高い
- **オフライン動作**: インターネット接続がなくても完全に機能する

#### なぜ技術的に難しいか

AquaVoiceが「単一話者のディクテーション」で成功を収めた一方、本プロダクトが取り組む課題は以下の点で技術的難易度が格段に高い。

| 観点 | AquaVoice（ディクテーション） | 本プロダクト（会議構造化） |
|------|---------------------------|------------------------|
| 話者数 | 1人 | 複数人（話者分離が必要） |
| 入力の性質 | 意図的な口述（明瞭） | 自然な議論（割り込み・曖昧表現） |
| 処理タスク | テキスト化のみ | テキスト化 + 論点抽出 + 構造化 + 追跡 |
| LLMの役割 | フィラー除去・整形 | 議論の意味理解・構造化・軌道修正判定 |
| リアルタイム性 | 発言→テキスト（1段階） | 発言→テキスト→構造化→UI更新（多段階） |
| コンテキスト長 | 短文（数十語） | 会議全体（数千〜数万語の累積コンテキスト） |

この技術的チャレンジの高さ自体が、競合に対する参入障壁となる。

## 差別化ポジション

```
                    会議後の振り返り ←──────────→ 会議中のリアルタイム支援
                         │                              │
  文字起こし+要約         │  Otter.ai / tl;dv / Rimo     │
  （コモディティ）         │  Google Meet / Teams / Zoom   │
                         │                              │
  リアルタイム要約         │                    SuperIntern│（Live Note）
                         │                              │
  アジェンダ管理          │  Fellow.app（手動作成前提）     │
                         │                              │
  ★論点構造化+           │                              │★ 本プロダクト
    軌道修正支援          │                     （空白地帯）│
                         │                              │
```

- **既存ツール**: 会議**後**のまとめ（事後的・受動的）、またはリアルタイムでも要約にとどまる
- **本ツール**: 会議**中**に論点を自動構造化し、議論の軌道修正まで行う（即時的・能動的）
- **ビッグテックとの差別化**: プラットフォーム横断 × 完全ローカル処理（Qwen3.5 + whisper.cpp） × OSSの組み合わせは、構造的にビッグテックが模倣できない領域
