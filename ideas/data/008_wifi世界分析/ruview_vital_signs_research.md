## 1. エグゼクティブサマリー

RuView（https://github.com/ruvnet/RuView）は、WiFi CSI（Channel State Information）を利用した非接触型バイタルサインモニタリングを謳うオープンソースプロジェクトである。呼吸数（6-30 BPM）と心拍数（40-120 BPM）のFFTベース検出を実装しており、高齢者見守り・転倒検知・睡眠時無呼吸検知等のユースケースを想定している。

**総合評価：**
- Rust実装のバイタルサイン検出コードは「実際の信号処理アルゴリズム」であり、プレースホルダーではない
- ただし、実環境での動作検証が極めて不十分で、独立した第三者による精度検証が存在しない
- プロジェクト全体の信頼性に対する強い懸念（AI生成コード疑惑、ドキュメント不整合、シミュレーションモードとの混同）
- 学術論文で実証されている精度水準（PulseFi: MAE 0.50 BPM@ESP32）と比較すると、RuViewの主張（±1 BPM）は検証されていない楽観的な数値
- **高齢者見守りの製品開発に直接使用するには時期尚早。研究参考・アーキテクチャ参考としての価値はある**

## 2. RuViewのバイタルサイン技術詳細

### 2.1 主張されている機能

| 機能 | 主張値 | 方式 |
|------|--------|------|
| 呼吸数検出 | 6-30 BPM、±1 BPM精度 | バンドパス 0.1-0.5 Hz → FFTピーク検出 |
| 心拍数検出 | 40-120 BPM | バンドパス 0.8-2.0 Hz → FFTピーク検出 |
| 転倒検知 | アラート遅延2秒以内 | モーション解析 |

| 歩行分析 | 移動パターン評価 | CSI変動解析 |
| 在室検知 | 95%以上の精度 | CSI振幅変動 |

### 2.2 コード実装の分析（Rust版）

**実コードの確認結果：**

RuViewのRust実装（`rust-port/wifi-densepose-rs/crates/wifi-densepose-vitals/`）には、以下のモジュールが存在する：

- `breathing.rs` - 呼吸数抽出（IIRバンドパスフィルタ + ゼロクロッシング検出）
- `heartrate.rs` - 心拍数抽出（IIRバンドパスフィルタ + 自己相関ベースピーク検出）
- `preprocessor.rs` - CSI前処理（指数移動平均フィルタによる静的成分除去）
- `anomaly.rs` - 異常検知（z-scoreベース）
- `store.rs` - 時系列データ保存
- `types.rs` - データ型定義

**コード品質評価：**

1. **呼吸検出（breathing.rs）**
   - 正規のIIR 2次共振器フィルタを実装
   - ゼロクロッシングレートによる周波数推定（標準的DSP手法）
   - 複数サブキャリアの加重平均による融合
   - 「最低10秒のデータ」等の実用的制約条件を実装
   - ランダム値やモックデータ生成なし
   - **評価：正規の信号処理コード**

2. **心拍検出（heartrate.rs）**
   - IIRバンドパスフィルタ（0.8-2.0 Hz）
   - 自己相関ベースのピーク検出
   - サブキャリア間の位相コヒーレンス重み付け
   - 生理学的パラメータに基づく設計（心臓信号は0.1-0.5mm変位、呼吸信号の1/10）
   - 信頼度スコアリング（自己相関ピーク強度ベース）
   - **評価：正規の信号処理コード**

3. **前処理（preprocessor.rs）**
   - 指数移動平均（EMA）フィルタ（α=0.05）
   - サブキャリアごとの静的環境モデリング
   - 残差計算（observed - predicted）で生理信号を抽出
   - **評価：正規の信号処理コード**

**重要な注意点：**
- コードは「信号処理アルゴリズムとしては正規」だが、これが実際のESP32 CSIデータで期待通りに動作するかは別問題
- テストケースは合成正弦波で検証しており、実環境CSIデータでの検証は確認できない
- Python v1実装には `amplitude=np.random.rand(3, 56)` のようなハードコーデッドランダム値が存在していた（Issue #37で指摘）

### 2.3 ADR-021：設計文書の分析

ADR-021（Architecture Decision Record）では8段階の信号処理パイプラインが定義されている：

1. CSI前処理（ESP32フレーム解析）
2. サブキャリア重み付け（Attention + GNN）
3. 呼吸抽出（バンドパス 0.1-0.5 Hz）
4. 心拍抽出（ModernHopfieldネットワーク）
5. アーティファクト除去（GlobalWorkspace競合）
6. 異常検知（Welford統計 + z-score + CUSUM）
7. コヒーレンスゲート（3フィルタパイプライン）
8. 階層化ストレージ（8bit/5bit/3bit精度）

**設計文書の目標精度：**
| 指標 | 目標値 |
|------|--------|
| 呼吸数誤差 | <1.5 BPM中央値 |
| 心拍数誤差 | <5 BPM中央値 |
| 検出遅延 | <15秒 |
| モーション除去 | >95%真陽性率 |
| 誤警報率 | <2% |

**注意：** これらは「目標値」であり、実測値ではない。ADR自体が「wellness-grade monitoring」であり「medical-grade」ではないと明記している。

## 3. プロジェクトの信頼性に関する重大な懸念

### 3.1 GitHub Issue #37：「これは偽物ではない」論争

- ユーザーから学術参照の誤引用が指摘（「WiFi CSI sensing survey, IEEE 2019」が実際にはビデオエンコーディングの論文）
- 「WiPose」論文がLoRaに関するものでWiFi姿勢推定ではないとの指摘
- Python v1コードに `np.random.rand()` によるハードコーデッドランダム振幅データが存在
- Dockerイメージが存在しない時期のドキュメントが公開されていた
- Redditでの批判投稿が削除された

### 3.2 GitHub Issue #79：「これは実用可能なプロジェクトか？」

- あるユーザーがWiFiアダプターを物理的に取り外しても「ハードウェアとスケルトンが表示され続けた」と報告
  - これはシミュレーションモードがデフォルトで動作しているため
- プロジェクトには3つのモード（simulated / wifi-based / ESP32 hardware）が存在し、混同されやすい
- ドキュメントで参照されている `docs/hardware-setup.md` が存在しない
- ビデオ証拠の提供要求に対して未回答

### 3.3 GitHub Issue #113：「大量のAI生成コンテンツで実行不可」

- ESP32ファームウェアの設定時に `nvs_partition_gen` モジュールが見つからないエラー
- ドキュメントに記載されている `--filter-mac` オプションがprovision.pyスクリプトに存在しない
- Docker起動後、UIエンドポイント（/ui/*）が全て404エラー
- Issueは「COMPLETED」として閉じられたが、問題は未解決のまま

### 3.4 GitHub Issue #51：「振幅ベース検出は非静的環境で脆弱」

- ESP32-S3ノードを実際にデプロイしたユーザーが、平均振幅測定が「ノイズに支配されている」と報告
- スペクトル分析で呼吸（0.1-0.5 Hz）や動作（0.5-3 Hz）の周波数帯にエネルギーが見られなかった
- **これは実環境での基本的なバイタルサイン検出の失敗を示す重要な証拠**

### 3.5 DeepWiki分析

DeepWikiによるコード分析では：
- `audit` ワーカー：44回実行、0回成功
- `optimize` ワーカー：34回実行、0回成功
- `testgaps` ワーカー：27回実行、0回成功
- 一部の重要インフラがスタブ/プレースホルダー実装である可能性

### 3.6 Forks分析

- フォークの中で高齢者ケアやバイタルサイン機能を独自に実装したものは確認できなかった
- 大半のフォークは未修正のコピーと推定される

## 4. 学術的検証との比較

### 4.1 PulseFi（UC Santa Cruz、2025年）— 最重要ベンチマーク

PulseFiはIEEE DCOSS-IoT 2025で発表された、ESP32 CSIベースの心拍数モニタリングシステム。

| 指標 | PulseFi（ESP32） | PulseFi（Raspberry Pi） | RuView主張値 |
|------|-------------------|------------------------|-------------|
| 心拍MAE | 0.50 BPM（5秒窓） | 0.17 BPM（30秒窓） | ±1 BPM（未検証） |
| 呼吸MAE | 0.09 breaths/min | - | ±1 BPM（未検証） |
| 臨床閾値内率（HR） | 97.95%（±1.5 BPM以内） | - | 記載なし |
| 臨床閾値内率（BR） | 99.6%（±0.75 BPM以内） | - | 記載なし |
| 無呼吸検出精度 | 99.4% | - | 記載なし |
| 被験者数 | 7人（ESP32）、118人（EHealth） | - | 0人（検証なし） |
| サブキャリア数 | 64（ESP32） | 234（RPi） | 56（ESP32-S3） |

**重要な知見：**
- ESP32の64サブキャリアでは、5秒窓でMAE 0.50 BPMが精度限界
- より長い測定窓でも改善しない（サブキャリア解像度がボトルネック）
- Raspberry Piの234サブキャリアでは3.6倍の周波数分解能により大幅に改善
- PulseFiは単一アンテナ振幅情報のみ使用（位相不要）でこの精度を達成

### 4.2 学術論文サーベイの精度水準

PMC（PubMed Central）の系統的レビュー結果：

**呼吸数検出：**
- 最高精度：>99%（BreathTrack、ノイズ低減+モデリングベース）
- リアルタイム遠距離：MAE < 0.3 BPM
- 中央値誤差：0.06-0.15 BPM
- 多人数時の劣化：1人96%→5人62%（0.5 BPM以内）

**心拍数検出：**
- 呼吸より大幅に困難（心臓変位0.1-0.5mm vs 呼吸変位1-5mm）
- 中央値誤差：1.19 BPM（最良条件）
- 推定誤差：<10 BPM（一般的条件）
- AI手法で95%以上の精度を達成した報告あり

**共通の限界：**
- 多人数環境での精度劣化が最大の課題
- 環境変動（壁材質、室内レイアウト、距離）への依存性
- 被験者の体位・動きによる不確実性
- 実験室環境と実環境のギャップが大きい
- 大半の研究は1-12人の少数被験者

### 4.3 PeerJの2025年系統的レビュー

- AI強化CSI WiFiセンシングのバイタルサイン監視レビュー
- 心拍モニタリング：95%以上の精度報告
- 呼吸数：81.8-99.97%の精度（条件により大きく変動）
- 睡眠段階分類：81.8%
- **実際の臨床実装は極めて限定的**
- 管理された実験環境と実際の介護環境のギャップが大きい

## 6. ESP32 CSIによるバイタルサイン検出の現実的な限界

### 6.1 物理的限界

1. **サブキャリア数の制約**：ESP32は64サブキャリア（S3は56サブキャリア）のみ。Raspberry Piの234サブキャリアと比較して周波数分解能が約1/4。これが心拍検出精度のハードリミットとなる。

2. **心臓変位の微小さ**：心拍による体表面変位は0.1-0.5mm。呼吸の1-5mmと比較して1桁小さく、SNRが本質的に低い。

3. **フレネルゾーン依存性**：検出品質はTX-RX間の第1フレネルゾーンに対する被験者の位置に強く依存。

### 6.2 環境的限界

1. **多人数問題**：単一TX-RXペアからの多人数バイタルサイン分離は未解決の研究課題。5人同時で呼吸検出精度が62%まで低下。

2. **通壁性能の劣化**：壁越しの検出はLOS（見通し線）と比較して大幅に性能が低下。

3. **環境変動への脆弱性**：家具の移動、ペットの存在、HVAC気流、電子機器の干渉等により誤検知が発生。

4. **キャリブレーション必須**：部屋ごとに15-60秒の初期キャリブレーションが必要。環境変化のたびに再キャリブレーションが求められる。

### 6.3 臨床適用の限界

1. **医療機器として不適格**：WiFi CSIベースのバイタルサインは統計的集約値であり、臨床診断が求める精度を提供できない。

2. **大規模検証の不在**：PulseFiの118人が最大規模。数千人規模の臨床試験は存在しない。

3. **高齢者特有の課題**：不整脈、低呼吸、体動の多さ等、高齢者特有の状態に対する検証データがほぼ存在しない。

## 8. 結論

### RuViewの評価

**ポジティブ面：**
- WiFi CSIバイタルサインセンシングの概念実証としてのアーキテクチャ設計は包括的
- Rust実装の信号処理コードは「アルゴリズム的には正規」
- 60のエッジモジュール構想は野心的
- プライバシー保護（カメラ不要）のアプローチは高齢者見守りに適している
- ハードウェアコスト（ESP32メッシュ約$54）は極めて低い

**ネガティブ面：**
- 実環境でのバイタルサイン検出の動作実績が確認できない
- 複数のGitHub Issueで根本的な動作問題が報告されている
- シミュレーションモードがデフォルトで動作し、実データとの混同が発生
- AI生成コード疑惑、ドキュメント不整合、参照論文の誤引用
- 独立した第三者による検証が一切存在しない
- ESP32-S3で実際にCSIデータ取得→バイタルサイン検出のe2e動作を確認した独立報告がない

### 最終判断

RuViewは「WiFi CSIバイタルサインセンシングのアーキテクチャ参考資料」としては価値があるが、「高齢者見守りシステムの実装基盤」としては信頼性が不十分である。高齢者見守りの製品開発を進める場合は：

1. **基盤技術**：Espressif ESP-CSI公式ツールキットから開始
2. **バイタルサイン手法**：PulseFi（UC Santa Cruz）の学術的に検証された手法を参考
3. **参考設計**：RuViewのADR-021パイプライン設計をアーキテクチャガイドとして活用
4. **精度期待値**：心拍MAE 0.5 BPM（ESP32、静止・単独・5秒窓）が現実的上限
5. **スコープ制限**：まずは在室検知+活動監視から開始し、バイタルサインは段階的に追加
