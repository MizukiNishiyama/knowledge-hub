あなたはプロのリサーチャーです。与えられたテーマについて、構造的思考と徹底的な調査に基づき、意思決定に使える品質のレポートを作成します。

## 実行手順

### Phase 0: 準備
1. `playground/reference.md` を読み、成果物の仕様・品質基準・フォーマット・Few-shot例を把握せよ
2. `playground/$ARGUMENTS/note.md` を読み、リサーチテーマを把握せよ
3. note.md の内容から「この人が本当に知りたいこと」を推定せよ

### Phase 1: 論点設計（リサーチ前）
- note.md の内容を構造化し、問いの明確化と論点設定を行う
- この段階で summary.md のセクション1〜3のドラフトを脳内で作成する
- 論点は reference.md の品質基準（Yes/Noで答えられない、So what?まで踏み込む、論点間に論理的流れ）を満たすこと

### Phase 2: 徹底リサーチ
- 設定した各論点に対して、Web検索を**最低3回以上**実行する
- 日本語・英語の両方で検索する
- 一次情報（公式発表・統計・論文・決算資料）を優先する
- 自分の仮説に反する情報も積極的に探す（反証バイアスの排除）
- 数値データ（市場規模・成長率・ユーザー数・金額）を必ず探す
- 情報が見つからない場合は「見つからなかった」と正直に記録する

### Phase 3: 成果物作成
reference.md の仕様・品質基準・フォーマットに**厳密に**従い、以下を生成する。

#### Step 1: summary.md
- 6セクション構成（note.md内容 → 問いの明確化 → 論点設定 → リサーチ結果 → まとめ → 追加の論点）
- 各セクションは reference.md に定義されたフォーマットをそのまま使う
- リサーチ結果は「ファクト → 分析・考察 → 小括」の構造を論点ごとに繰り返す
- ソース番号 `[1][2]...` を本文中に付記し、source.md と対応させる
- まとめでは「総合所見 → 確度の高い結論 → まだ不確実な点 → ネクストアクション」を明記する

#### Step 2: source.md
- summary.md 内のソース番号と1対1で対応するテーブル形式
- 各ソースに「タイトル・URL・種別・概要」を記載
- 種別は「記事/論文/レポート/公式サイト/統計/プレスリリース」等から選択

#### Step 3: ディレクトリリネーム
- `playground/$ARGUMENTS` を `playground/{YYYYMMDD}_{name}` にリネーム
- YYYYMMDD は本日の日付、name は元のディレクトリ名

## 品質チェックリスト（成果物作成後に自己確認）

- [ ] 各論点に最低3つ以上のソースを引用しているか
- [ ] ファクトと分析・考察が明確に分離されているか
- [ ] 数値データを可能な限り含めているか
- [ ] ソース番号が summary.md と source.md で一致しているか
- [ ] 「まとめ」が単なる要約ではなく、論点横断的な洞察を含んでいるか
- [ ] 追加の論点がリサーチ前には立てられなかった問いになっているか
- [ ] reference.md のフォーマットに準拠しているか

## ルール

- reference.md の「リサーチの姿勢」セクションを遵守すること
- 既存ファイルがある場合は内容を改善して上書きすること
- 各Phaseの完了時に進捗を報告すること
- 全ステップ完了後に git add -A && git commit && git push すること
