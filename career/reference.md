# Career リファレンス

## 概要

`career/` ディレクトリは、西山瑞生のキャリアについて深く考えるための作業空間。
思考のメモから論点整理・リサーチ・サマリー生成までを一気通貫で行い、常に最新のキャリアプランを維持する。

---

## ディレクトリ構造

```
career/
├── reference.md              # 本ファイル。仕様・運用ルールを定義
├── {YYYYMMDD}_latest_plan.md # 常に最新のキャリアプラン（1ファイルのみ存在）
├── archive/                  # 過去の latest_plan のアーカイブ
│   └── {YYYYMMDD}_latest_plan.md
└── thoughts/                 # 各問い・論点を格納するディレクトリ
    └── {YYYYMMDD}_{問いの超簡潔な要約}/
        ├── note.md           # ユーザーが書いたメモ（原文）
        ├── summary.md        # リサーチ・論点整理の成果物
        └── summary.pdf       # summary.md のPDF版
```

---

## トリガー：/career コマンド

1. `career/thoughts/` 配下の最新ディレクトリの `note.md` を読む
2. `career/{YYYYMMDD}_latest_plan.md` を読む
3. note.md を分析・リサーチし、`summary.md` と `summary.pdf` を生成する
4. 追加論点候補が出たらユーザーに確認する
5. Yes の場合、latest_plan を更新する

---

## summary.md の仕様

### 8章構成

---

### # 問い（原文）

note.md の原文をブロック引用（`>`）でそのまま掲載。一切の編集を加えない。

---

### # 問い（フォーマット後）

**目的**：散発的なメモを調査可能な形に構造化する

**品質基準**：
- note.md と latest_plan を突き合わせ、「なぜ今この問いを持つのか」を推定する
- 暗黙の前提・仮説を明示する
- 曖昧な表現を具体的な問いに変換する

**フォーマット**：
```markdown
### 背景・文脈
（note.md と latest_plan から読み取れる背景を2-3文。「今このタイミングでこの問いを持つ理由」を含める）

### 核心的な関心事
- （最も知りたいことを1-3個、明確な問いの形で）

### 暗黙の前提・仮説
- （note.md の記述から推定される前提を列挙）
```

---

### # 結論

**目的**：リサーチ全体を総合し、西山瑞生の意思決定に直結する示唆を導く

**品質基準**：
- 単なる要約ではなく、**論点横断的な洞察**を含める
- 「結局どういうことか」を**3文以内**で言い切る
- 確度の高い結論とまだ不確実な部分を明確に区別する
- 「だから次に何をすべきか」という具体的なネクストアクションを含める

**フォーマット**：
```markdown
### 総合所見
（リサーチ全体を通じた結論を3文以内で）

### 確度の高い結論
- （根拠が十分にある結論。ソース番号を付記）

### まだ不確実な点
- （追加調査・経験が必要な点）

### ネクストアクション
- （具体的に次にやるべきこと。期限・方法まで踏み込む）
```

---

### # 論点整理

**フォーマット**：
```markdown
| # | 論点 | 調査の方向性 | latest_planとの関連 |
|---|------|-------------|-------------------|
```

**品質基準**：
- 3〜6個の論点を設定する
- 各論点は「Yes/Noで答えられない、調査と考察を要する問い」にする
- latest_plan のパスと接続する論点を少なくとも1つ含める

---

### # リサーチ結果

**フォーマット（論点ごとに繰り返し）**：
```markdown
### 論点N: （論点をそのまま再掲）

#### ファクト
- （調査で判明した事実。ソース番号を付記 [1][2]）
- （数値データがあれば明記）

#### 分析・考察
（ファクトを踏まえた解釈。キャリア文脈での意味合いを含む。複数の見方がある場合は併記）

#### 小括
（この論点に対する暫定的な回答を1-2文で。西山瑞生への示唆を含む）
```

**品質基準**：
- 各論点につき最低3つ以上のソースを参照する
- ファクトと分析を明確に分離する
- 数値データを必ず含める
- 相反する見解がある場合は両論併記する

---

### # 説明

**目的**：結論に至った思考プロセスの補足。なぜその結論が導かれるかのロジックを示す

**品質基準**：
- 複数の論点を横断的に接続し、全体として何を意味するかを語る
- latest_plan との整合性・矛盾点を明示する
- 反証をどう解釈したかを明示する
- 「なぜ別の結論にならなかったのか」を1つ以上説明する

---

### # ソース

**フォーマット**：
```markdown
| # | タイトル | URL | 種別 | 概要 |
|---|---------|-----|------|------|
```

種別：記事 / 論文 / レポート / 公式サイト / 統計 / 採用情報 / インタビュー

---

### # 追加論点候補

**目的**：リサーチを踏まえて新たに浮上した問いを提示し、latest_plan との接続を示す

**品質基準**：
- リサーチ前には見えなかった問いにする
- 2〜4個程度
- latest_plan のどのパスに関係するかを明示する

**フォーマット**：
```markdown
| # | 追加論点 | latest_planへの関連・なぜ重要か |
|---|---------|-------------------------------|
```

## latest_plan.md の仕様

### ファイル仕様
- ファイル名：`career/{YYYYMMDD}_latest_plan.md`（日付は更新日）
- 常に1ファイルのみ存在する

### 6章構成（冒頭に「# なぜこのキャリアを志すのか」を必ず置く）

---

### # なぜこのキャリアを志すのか

**目的**：このキャリアを追う根本的な動機を明文化し、判断に迷ったときの原点として機能させる

**品質基準**：
- 「AGIの到来」という外部環境の変化と「自分がなぜそこでバリューを出せるか」の両方を含める
- 以下の3層構造で書く：①前提（AGIが社会に何をもたらすか）、②なぜ国会議員でなければならないか、③自分がそのポジションに到達できる根拠
- AGIがもたらす具体的な国家論点（UBI・雇用・AGI所有権・国防・産業構造等）を箇条書きで列挙する
- 更新のたびに内容を見直す必要はない。根本動機が変わった場合のみ更新する

**フォーマット**：
```markdown
## 前提：AGIは〇〇年代に到来し、社会の基本設計を壊す
（AGI到来の時期と社会的インパクトを2〜3文で）

## AGIがもたらす国家レベルの論点群
- （論点を箇条書きで。UBI・雇用・AGI所有権・国防・経済・教育等）

## なぜ「国会議員」でなければならないか
（実業家提言型・官僚型・国会議員型の3択を比較し、国会議員型を選ぶ理由を論じる）

## 自分がこのポジションに到達できる根拠
（現在・2029年時点のアセットを列挙。希少性を明示する）
```

### # キャリアパス概要

**目的**：現在検討中のキャリアパスの全体地図を一覧できるようにする

**フォーマット**：
```markdown
| パス名 | 一言説明 | 魅力 | 難易度 | 確度 |
|--------|---------|------|--------|------|
```

その下に：
```markdown
### パス間の関係性
- （排他・補完・シーケンシャルの関係を記述）
```

**品質基準**：
- 検討中のパスを2〜5個列挙する
- 確度（高/中/低）と難易度（高/中/低）を更新のたびに見直す

---

### # キャリアパス詳細

**目的**：各パスについて「具体的に何をするか」と「何年後にどうなるか」を明確にする

**フォーマット（パスごとに繰り返し）**：
```markdown
### パスN: 〇〇

#### 道筋
- 入口：（現在地→このパスに入るための具体的な一歩）
- 中間：（2〜3年後の状態）
- 到達点：（5年後の理想状態）

#### 必要なアクション
- 今すぐ（〜1ヶ月）：
- 3ヶ月以内：
- 1年以内：

#### タイムライン
- 〇年後：（マイルストーン）

#### 成功の定義
- 定量：（年収・役職・実績数など）
- 定性：（どんな仕事をしていて、どんな人と関わっているか）
```

**品質基準**：
- 抽象的な表現を避け、具体的な職種名・企業名・スキル名・金額で書く
- 「今すぐ動けるアクション」を必ず1つ以上含める

---

### # 各パスの理論的裏付け

**目的**：「なぜ自分にこのパスが合っているか」を感覚ではなくロジックで言語化する

**フォーマット（パスごとに繰り返し）**：
```markdown
### パスN: 〇〇

#### 強み軸
（自分の経験・スキル・特性がこのパスでどう活きるか）

#### 興味軸
（なぜこのパスに惹かれるのか。内発的動機の根拠）

#### 市場軸
（このパスの需給・成長性・競合状況。数値があれば引用）

#### 最大のリスクと対策
- リスク：（失敗する最も可能性の高いシナリオ）
- 対策：（リスクを下げるために今できること）

#### 選ばない理由（もし選ばないとしたら）
（このパスを捨てるとしたらどんな理由か。1文で）
```

## latest_plan 更新オペレーション

1. ユーザーが「yes」と回答
2. 現在の `{YYYYMMDD}_latest_plan.md` を `career/archive/` にコピー（ファイル名そのまま）
3. 追加論点・示唆を反映した新しい latest_plan を上記仕様で作成
4. ファイル名のプレフィックスを**本日の日付**に変更して `career/` 直下に保存
5. `career/` 直下の旧ファイルを削除（archive に移行済みのため）

---

## リサーチの姿勢

- **一次情報を優先する**：公式発表・統計・論文・採用情報・キャリア事例を重視する
- **数字で語る**：「〜が多い」ではなく「〜%」「〜件」「〜万円」で示す
- **反証を探す**：自分の仮説に都合の悪い情報も積極的に探す
- **日英両方で検索する**：日本固有のテーマでも英語圏に有益な分析があることが多い
- **「なぜ？」を繰り返す**：表面的な事実の裏にある構造・メカニズムまで掘り下げる
- **キャリア文脈を忘れない**：リサーチの目的は「西山瑞生の意思決定に使えること」
